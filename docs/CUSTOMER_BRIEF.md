# Brief для замовника: план реалізації бізнес-процесу

## 1. Мета проєкту
Побудувати стабільний сервіс, який автоматично:
- отримує замовлення з CRM на етапі `Матеріали`;
- формує коректні макети (poster/engraving/sticker) за правилами ТЗ;
- враховує ручні матеріали (`A6`, `брелок`) у загальній нумерації;
- додає QR або Spotify code тільки для визначених товарів;
- відправляє матеріали в Telegram-гілку `ОБРОБКА`;
- змінює статуси замовлення в CRM за Telegram-реакціями.

Ключовий пріоритет: передбачувана робота без ручних “доправок” і без втрати замовлень у разі збоїв зовнішніх сервісів.

## 2. Бізнес-процес (цільовий потік)
### 2.1 Прийом події з CRM
Вхід:
- webhook від CRM по Shopify-замовленню.

Перевірки:
- валідний webhook secret;
- коректна структура payload;
- idempotency ключ (щоб не обробляти дублікати).

Результат:
- подія приймається і ставиться в `order_intake` чергу;
- HTTP-відповідь повертається швидко, без важкої обробки.

Поведінка при помилці:
- невалідний secret -> запит відхиляється;
- дубль події -> без повторної обробки.

### 2.2 Завантаження і нормалізація замовлення
Вхід:
- `order_id` з webhook.

Дії:
- довантаження повної картки замовлення через API CRM;
- перевірка, що замовлення на цільовому статусі `Матеріали`;
- нормалізація полів товарів, властивостей, посилань, кастомних опцій.

Результат:
- формується єдиний нормалізований об’єкт замовлення для всіх наступних кроків.

Поведінка при помилці:
- тимчасові CRM помилки -> retry/backoff;
- після вичерпання спроб -> DLQ + технічний alert.

### 2.3 Побудова layout-плану
Вхід:
- нормалізоване замовлення з CRM.

Дії:
- визначення матеріалів по товарах: `poster`, `engraving`, `sticker`;
- визначення ручних позицій (`A6`, `брелок`) для включення в `total` нумерації;
- визначення формату, типу підставки, special-коду товару;
- формування імен файлів за шаблоном ТЗ;
- визначення прапорів: `QR +`, `LF +`, `A6 +`, `B +`, `T`;
- перевірка посилання для QR/Spotify сценаріїв.

Результат:
- структурований `layout plan`:
  - повний список файлів;
  - порядок;
  - індексація;
  - примітки для Telegram.

### 2.4 Генерація PDF-комплекту
Вхід:
- `layout plan`.

Дії:
- генерація poster;
- генерація engraving (якщо в замовленні є текст гравіювання):
  - файл гравіювання у форматі `A3`;
  - межі тексту знизу підставки:
    - `A4`: `22x210 мм`;
    - `A5`: `22x148 мм`;
- генерація sticker (якщо є опція стікера, розмір фіксований);
- завантаження вихідних файлів, шрифтів і ресурсів з контрольованими timeout/retry.

Fallback-правила:
- якщо для engraving або sticker відсутній текст, blank PDF не створюється;
- комплект продовжує обробку без цього файлу;
- у Telegram додається явне попередження з ім'ям файлу, який не був згенерований.

Результат:
- підготовлений комплект PDF у службовому сховищі.

Поведінка при помилці:
- якщо PDF згенерувати неможливо -> задача в DLQ;
- для бізнесу фіксується статус `Без файлу`.

### 2.5 Вбудовування QR або Spotify code
Вхід:
- poster-файли з комплекту;
- SKU товару;
- посилання з замовлення.

Дії:
- перевірка, чи входить SKU в whitelist груп, де код взагалі дозволений;
- скорочення посилання перед QR:
  - primary: `lnk.ua`;
  - fallback: `cutt.ly`;
- вибір стратегії:
  - `spotify_code` для Spotify SKU + Spotify URL;
  - `qr` для інших релевантних сценаріїв;
  - `none` для товарів поза whitelist;
- для Spotify-сценарію:
  - shortener НЕ застосовується;
  - використовується Spotify URI/Spotify URL напряму;
  - завантаження SVG коду;
  - видалення фонового шару;
  - вставка в poster;
  - після вставки повторна заміна білого на off-white.

Результат:
- poster із вбудованим кодом у правильній позиції або коректна примітка, якщо код не треба вбудовувати.
- якщо QR замовлено, але код не можна вбудувати (unsupported SKU, missing `SKU/format`, invalid URL), оператор бачить явне `🚨`-попередження;
- `Посилання QR` показується тільки тоді, коли код реально був вбудований у макет.

### 2.6 Друкарська підготовка
Дії:
- заміна білого кольору на off-white для друку;
- CMYK-конверсія фінального PDF.

Результат:
- стабільний друкарський профіль макетів перед відправкою в Telegram.

### 2.7 Доставка в Telegram
Вхід:
- згенеровані файли;
- прев’ю;
- службові прапори і примітки.

Дії:
- надсилання прев’ю в гілку `ОБРОБКА`;
- надсилання PDF-файлів у гілку `ОБРОБКА`;
- додавання caption із переліком файлів і примітками.

Результат:
- команда отримує повний комплект для перевірки та запуску в роботу.

Структура Telegram простору:
- `ОБРОБКА`: автоматичний потік від бота;
- `ЗАМОВЛЕННЯ`: відібрані/підтверджені замовлення;
- `ЧАТ`: обговорення команди, без автоматичних дій бота.

Поведінка після реакції:
- `1 ❤️` змінює статус на `Друк` і запускає пересилання комплекту в `ЗАМОВЛЕННЯ`.

Важливий нюанс:
- помилка відправки прев’ю не повинна блокувати відправку PDF-файлів;
- критичним вважається саме недоставка файлів.
- якщо warning-и є, вони стоять на початку caption;
- якщо warning-ів немає, caption починається з `✅`.

Поведінка при помилці:
- недоставка файлів -> DLQ + статус `Немає в тг` + alert у технічний чат.

### 2.8 Прив’язка Telegram-повідомлень до замовлення
Дії:
- після успішної доставки зберігається mapping:
  - `chat_id + message_id -> order_id`.

Результат:
- система може однозначно визначити, до якого замовлення належить реакція в Telegram.

### 2.9 Реакції і зміна статусів у CRM
Вхід:
- reaction webhook з Telegram.

Дії:
- підрахунок кількості релевантних реакцій за workflow-emoji;
- визначення етапу workflow;
- застосування монотонного переходу (тільки вперед):
  - `1 ❤️` -> `Друк` + пересилання в `ЗАМОВЛЕННЯ`;
  - `👍` -> `Пакування` (пілотний сценарій, за замовчуванням вимкнений);
- дедуплікація повторних reaction-апдейтів.

Результат:
- автоматична зміна статусів у CRM без ручного втручання.

Важливий нюанс:
- rollback при знятті реакцій не виконується;
- повторні апдейти в межах того ж етапу ігноруються.
- для пілотного `👍`:
  - якщо `❤️` не було, цільова логіка: `Друк -> Пакування`;
  - якщо `Друк` вже стоїть, `👍` веде одразу в `Пакування`.

## 3. Правила, які мають бути дотримані
### 3.1 Неймінг і порядок матеріалів
- Формат імен згідно ТЗ (`CGU_...`), включно з `_T` для термінових.
- Порядок для комбінації `poster + engraving + sticker` фіксований:
  - `1`: poster
  - `2`: engraving
  - `3`: sticker
- Якщо в одному замовленні кілька poster, нумерація йде послідовно по порядку.
- Якщо в замовленні є ручні позиції (`A6`, `брелок`), вони входять у `total`.
- Шаблони ручних позицій:
  - `CGU_B_<order>_<i>_<total>`
  - `CGU_AA6_<order>_<i>_<total>`
- У Telegram caption блок прапорів уніфікований і показується як `📌 QR +`, `📌 LF +`, `📌 A6 +`, `📌 B +`.
- Файл гравіювання завжди формується як `A3` з межами:
  - `A4`: `22x210 мм`
  - `A5`: `22x148 мм`
- Розмір стікера фіксований.
- Якщо замовлений файл не згенеровано через відсутні вхідні дані, його порядковий номер все одно резервується в `total`, а в caption додається warning.

### 3.2 QR/Spotify логіка
- Код додається тільки для визначених SKU-груп.
- Для Spotify-груп:
  - Spotify-посилання -> Spotify code;
  - не-Spotify посилання -> звичайний QR.
- Для інших товарів код у PDF не додається, але прапори в примітках (`QR +`, `LF +`, `A6 +`, `B +`) зберігаються.
- Якщо в замовленні є QR, але SKU не входить у whitelist або для нього не вистачає технічних даних (`SKU`, формат), замовлення не блокується:
  - комплект макетів генерується без QR;
  - у Telegram додається явне червоне попередження `🚨`, що QR не згенеровано і не вбудовано в макет.

### 3.3 Підготовка до друку
- Для всіх фінальних макетів застосовується заміна білого на друкарський off-white.
- Для production-процесу використовується CMYK-конверсія.
- Preview не може використовуватись як джерело для друкарського postera.
- Якщо відсутній реальний design/source file, замовлення має йти у `Без файлу`.

### 3.4 Telegram workflow
- Бот підтримує і `group`, і `supergroup` (з `message_thread_id` для topic).
- Первинна доставка завжди йде в `ОБРОБКА`.
- Після `1 ❤️` виконується пересилання в `ЗАМОВЛЕННЯ`.
- `👍` зафіксований як пілотний відкладений етап і неактивний за замовчуванням.

## 4. Відмовостійкість і контроль помилок
### 4.1 Принцип
Webhook не виконує важку обробку напряму. Важкі операції виконуються у воркерах через черги.

### 4.2 Що робимо при збоях
- CRM/Telegram/Spotify/source URL помилки:
  - timeout + retry + backoff;
  - після вичерпання спроб задача переходить у DLQ.
- Якщо немає PDF:
  - замовлення переводиться у статус `Без файлу`.
- Якщо доставка в Telegram не відбулася:
  - замовлення переводиться у статус `Немає в тг`.
- Критичні збої:
  - логуються;
  - дублюються в технічний Telegram-чат.

### 4.3 Стійкість стану
Операційний стан зберігається в PostgreSQL:
- idempotency ключі webhook;
- message mapping Telegram -> order;
- workflow стан реакцій;
- dead letters (DLQ).

Це дозволяє безпечно перезапускати сервіс і не втрачати контроль процесу.

## 5. Логування, моніторинг, контроль
### 5.1 Логування
Передбачаються структуровані логи по ключових етапах:
- intake webhook;
- enqueue/dequeue;
- CRM fetch/update;
- PDF pipeline start/finish/error;
- QR/Spotify embed;
- Telegram delivery;
- reaction workflow;
- retry/DLQ/alerts.

### 5.2 Технічні сигнали
- `/health` для швидкої перевірки стану сервісу.
- Окремі ops-alert повідомлення в Telegram для інцидентів.

## 6. План перевірки якості
### 6.1 Автоматичні перевірки
- unit/integration тести:
  - naming і layout rules;
  - QR/Spotify decision logic;
  - queue retry/DLQ;
  - CRM/Telegram adapters;
  - reaction workflow.

### 6.2 Ручний UAT
- перевірка на реальних тестових замовленнях:
  - всі типові комбінації матеріалів;
  - кейс із ручними `A6/B` у загальній нумерації;
  - термінові замовлення;
  - QR/Spotify сценарії;
  - shortener fallback (`lnk.ua -> cutt.ly`);
  - невалідні посилання;
  - реакція `1❤️` + пересилання в `ЗАМОВЛЕННЯ`;
  - перевірка, що `👍` неактивний у дефолтному режимі.

## 7. План запуску в production
1. Узгодження фінальних правил і конфігурації.
2. Налаштування webhook у CRM і Telegram.
3. Прогін smoke/UAT у тестовому контурі.
4. Запуск у production.
5. Моніторинг інцидентів і контроль DLQ у перший період роботи.

## 8. Що потрібно від замовника для завершення
- Фінальні доступи і значення для `env`:
  - CRM token/base URL;
  - Telegram bot token + chat/thread для `ОБРОБКА`, `ЗАМОВЛЕННЯ`, ops-чату;
  - webhook secrets;
  - DB доступ.
- Підтвердження бізнес-правила по реакціях:
  - чи враховуємо реакції тільки на PDF-повідомленнях, чи також на preview.
- Підтвердження пілоту:
  - коли вмикаємо `👍 -> Пакування`;
  - чи потрібна деталізована картка замовлення у повідомленні перед пакуванням.
- Підтвердження політики пересилання:
  - `copy` (рекомендовано) чи `forward`;
  - чи дублювати після `1 ❤️` в приватний чат менеджера.
- 2-3 тестових замовлення для фінального UAT.

## 9. Очікуваний результат для бізнесу
- Менше ручної рутини в обробці макетів.
- Менше помилок у файлах і статусах.
- Прозорий, контрольований і масштабований процес.
- Швидке виявлення проблем завдяки логам, алертам і DLQ.
