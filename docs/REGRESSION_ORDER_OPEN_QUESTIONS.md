# Regression Order Open Questions

## Призначення
Цей документ фіксує відкриті питання, неточності і неоднозначності по regression-набору тестових замовлень.

Це не decision-log і не кодова специфікація. Це робочий список для owner-а, поки не прийняті остаточні бізнес-рішення.

Актуальний зріз:
- дата: `2026-04-02`
- набір замовлень: `29061-29075`
- джерело: live dry-run по KeyCRM payload

Загальна примітка:
- усі замовлення з цього набору зараз мають статус `50`, а не `20`;
- це не бізнес-помилка payload, але через це реальний webhook-flow не підхопить їх без переведення на `Матеріали`.

## Як читати цей список
- якщо по лоту написано `відкритих питань немає`, це означає, що на рівні поточної бізнес-логіки він виглядає зрозумілим;
- якщо питання є, це не означає, що замовлення “битe”, а тільки те, що частина правила ще не формалізована в документації.

### 29068
Сценарій:
- `Two-base-poster order with Live Photo, keychain, mixed engraving/sticker, urgent`

Відкриті питання:
- відкритих питань наразі немає.

Що вже виглядає зрозумілим:
- multi-poster як тип замовлення дозволений;
- engraving і sticker можуть належати різним базовим poster в межах одного order.
- `Live Photo` не входить у перелік друкованих файлів і лишається тільки в замітках як `LF +`.
- `брелок` входить у `total` як ручний друкований файл і додається вручну наприкінці комплекту.
- source/design для ручного брелка не генерується системою і не вибирається автоматично: це ручна операція менеджера/оператора поза PDF pipeline.
- `urgent` у цьому кейсі поширюється на весь комплект друкованих файлів.
