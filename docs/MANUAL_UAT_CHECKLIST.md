# Manual UAT Checklist (Owner)

Це список перевірок, які потрібно виконати вручну перед production switch.

## 1. PDF і макети (Stage F)
1. Замовлення `лише poster`:
   - файл згенеровано;
   - назва файлу правильна;
   - у PDF немає чистого білого артефактного фону;
   - QR/Spotify (якщо є) в правильній позиції.
2. Замовлення `poster + engraving`:
   - обидва файли присутні;
   - порядок імен і індекси коректні (`1_2`, `2_2`);
   - гравіювання читається, шрифт коректний;
   - файл гравіювання згенеровано в `A3`;
   - текст у межах зони:
     - для `A4` підставки: `22x210 мм`;
     - для `A5` підставки: `22x148 мм`.
3. Замовлення `poster + engraving + sticker`:
   - 3 файли в правильному порядку;
   - назви з правильним `total=3`;
   - стікер згенеровано без обрізань.
4. Перевірка термінового:
   - в назвах є суфікс `_T`.
5. Перевірка special SKU:
   - коди типу `HT/RB/...` відповідають ТЗ.
6. Перевірка ручних позицій:
   - кейс `poster + A6 + брелок` має правильний `total=3`;
   - poster має індекс `1_3`;
   - ручні позиції мають `2_3` і `3_3`.

## 2. Telegram доставка (Stage G)
1. На статусі `Матеріали` файли йдуть у гілку `ОБРОБКА`.
2. На інших статусах відправка не відбувається.
3. У повідомленні є прев'ю (`_customization_image`), якщо воно було.
4. У підписі є `QR +` / `LF +` / `A6 +` / `B +` за потреби.
5. При помилці URL є примітка про невалідне посилання.
6. У БД (`telegram_message_map`) з'являється зв'язок `message -> order`.
7. Бот коректно працює в group і supergroup (з topic/thread).
8. shortener fallback працює:
   - якщо `lnk.ua` недоступний, використовується `cutt.ly`.

## 3. Реакції і статуси CRM (Stage H)
1. Поставити `1 ❤️` під повідомленням:
   - статус замовлення в CRM стає `Друк`.
2. Після `1 ❤️`:
   - комплект пересилається/копіюється в гілку `ЗАМОВЛЕННЯ`.
3. Додати `👍`:
   - у поточному релізі новий статус не застосовується (`PACKING pilot disabled`).
4. Зменшити реакції назад:
   - rollback не відбувається (статус не знижується).
5. Дубльовані/масові апдейти реакцій:
   - немає фліп-флопу статусів;
   - немає лавини повторних задач.

## 4. Відмовостійкість і recovery (Stage I)
1. Перевірити deterministic `Без файлу` без PDF/DLQ:
   - order без `_tib_design_link_1` або без тексту engraving/sticker;
   - PDF-повідомлення в `ОБРОБКА` не з'являються;
   - у CRM ставиться `Без файлу` (`40`);
   - у processing і ops chat приходить `error` alert;
   - запису в `dead_letters` немає.
2. Перевірити deterministic `source unavailable` без PDF/DLQ:
   - `_tib_design_link_1` є, але CDN/TeeInBlue віддає `403/404`;
   - PDF-повідомлення в `ОБРОБКА` не з'являються;
   - CRM статус не змінюється;
   - у processing і ops chat приходить `Не вдалося сформувати PDF`;
   - запису в `dead_letters` немає.
3. Зімітувати збій PDF після старту pipeline:
   - job потрапляє в таблицю `dead_letters`;
   - у CRM ставиться `Без файлу` (`40`);
   - у ops-чат приходить alert.
4. Зімітувати збій Telegram delivery:
   - job у DLQ;
   - у CRM ставиться `Немає в тг` (`59`);
   - у ops-чат приходить alert.
5. Перезапуск сервісу:
   - `/health` повертає `ok`;
   - нові webhook обробляються штатно;
   - message-map/idempotency/DLQ не губляться в БД.
6. Retention cleanup:
   - старі файли з `OUTPUT_DIR` і `TEMP_DIR` видаляються за політикою.

1. Усі кроки вище пройдено без блокуючих дефектів.
2. Немає нових DLQ записів після стабільного прогону тестових замовлень.
3. Ops-алерти не показують постійних критичних помилок.
