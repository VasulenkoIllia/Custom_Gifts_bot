# Додатковий план доробки (поза стартовим ТЗ)

## 1. Призначення
Цей документ фіксує нові вимоги, які були додані після стартового ТЗ:
- робота бота в структурі Telegram із гілками `ОБРОБКА / ЗАМОВЛЕННЯ / ЧАТ`;
- пересилання (copy/forward) матеріалів у цільові чати/гілки за бізнес-подією;
- включення ручних матеріалів (`A6`, `брелок`) у загальну нумерацію комплекту;
- скорочення посилань через shortener (`lnk.ua` пріоритетно, fallback `cutt.ly`);
- оновлена реакційна логіка:
  - `1 ❤️` -> статус `Друк` + пересилання в `ЗАМОВЛЕННЯ`;
  - `👍` -> статус `Пакування` (пілотний/відкладений етап до окремого рішення).

## 2. Узгоджені правила
1. Основний потік матеріалів із сайту потрапляє в гілку `ОБРОБКА`.
2. Після `1 ❤️` під релевантним повідомленням:
   - змінюється статус замовлення в CRM на `Друк`;
   - виконується пересилання/копіювання матеріалів у `ЗАМОВЛЕННЯ` (або іншу ціль, якщо так задано конфігом).
3. `👍` (пакування) лишається відкладеним пілотним етапом:
   - якщо `❤️` не було, але є `👍`, цільова логіка: послідовно `Друк -> Пакування`;
   - ввімкнення цього сценарію окремо підтверджується після UAT з командою.
4. Бот має працювати як у звичайній групі, так і в супергрупі (включно з topic/thread).
5. Якщо в замовленні є `A6` або `брелок`, вони додаються в `total` нумерації:
   - poster: `CGU_AA5_<order>_1_3`
   - брелок (ручний): `CGU_B_<order>_2_3`
   - A6 (ручний): `CGU_AA6_<order>_3_3`

## 3. Ключові рішення по реалізації
### 3.1 Топологія Telegram
- вводимо явний конфіг топології:
  - `processing` (де бот публікує макети автоматично);
  - `orders` (куди йдуть відібрані/підтверджені замовлення);
  - `chat` (для обговорення, без автоматики бота);
- цілі мають підтримувати:
  - `chat_id`;
  - optional `message_thread_id`.

### 3.2 Механіка пересилання
- рекомендується `copyMessage/copyMessages` як базова стратегія;
- fallback на `forwardMessage`, якщо для конкретного кейсу це потрібно;
- пересилання виконується окремою чергою, не блокує webhook intake.

### 3.2.1 Shortener політика
- перед генерацією QR посилання скорочується;
- primary: `lnk.ua`;
- fallback: `cutt.ly`;
- якщо shortener недоступний, діє retry/backoff і технічне попередження в ops-чат.

### 3.3 Реакційний workflow
- реакції аналізуються тільки на повідомленнях, прив’язаних до замовлення через message-map;
- stage `PRINT` (`1 ❤️`) активний;
- stage `PACKING` (`👍`) позначений як pilot/disabled за замовчуванням;
- rollback при знятті лайків не виконується.

### 3.4 Контроль дублювань
- ідемпотентність на рівні:
  - webhook;
  - reaction stage bucket;
  - forwarding event (`order + stage + target`).

## 4. Поетапний план реалізації
## Етап J1. Telegram topology config
Завдання:
- додати конфіг для `ОБРОБКА / ЗАМОВЛЕННЯ / ЧАТ`;
- валідувати режими group/supergroup/thread.

Критерій:
- сервіс стартує тільки з валідною topology-конфігурацією.

## Етап J2. Routing/forwarding service
Завдання:
- реалізувати сервіс пересилання макетів у цільові чати/гілки;
- додати retry/backoff/DLQ для forwarding pipeline;
- додати логування і ops alert на помилки пересилання.

Критерій:
- при ручному виклику routing подія стабільно доставляє файли у цільову гілку.

## Етап J3. Реакція `1 ❤️` -> `Друк` + пересилання
Завдання:
- підв’язати routing після успішної зміни статусу на `Друк`;
- пересилати тільки релевантні файли (poster/engraving/sticker) без службового шуму;
- зафіксувати антидублювання.

Критерій:
- один reaction stage викликає рівно одну зміну статусу і рівно одну доставку в `ЗАМОВЛЕННЯ`.

## Етап J4. Пілотний етап `👍 -> Пакування`
Завдання:
- тримати stage `PACKING` вимкненим за замовчуванням;
- підготувати режим, де `👍` виконує:
  - `Друк -> Пакування`, якщо `❤️` ще не було;
  - лише `Пакування`, якщо `Друк` вже стоїть.

Критерій:
- `👍` не змінює статус у дефолтному релізі, але може бути увімкнений конфігом для пілота.

## Етап J5. QA і запуск
Завдання:
- автотести на routing/reaction/ідемпотентність;
- ручний UAT по реальних замовленнях;
- smoke перевірка в production-контурі.

Критерій:
- стабільний цикл `Матеріали -> Друк -> пересилання в ЗАМОВЛЕННЯ` без дублювань і без втрати файлів.

## 5. Потрібні уточнення перед код-етапом
Рекомендовані дефолти для старту реалізації:
1. Реакції workflow враховуємо лише на повідомленнях із PDF-файлами (preview ігноруємо).
2. Для `1 ❤️` пересилаємо весь комплект матеріалів, включно з ручними `A6/B` позиціями у total-індексації.
3. Базова політика пересилання: `copy` (без згадки про джерело), fallback `forward` як аварійний режим.
4. `👍 -> Пакування` тримаємо disabled до окремого пілотного ввімкнення.
